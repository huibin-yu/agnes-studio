| 优先级 | 含义 | 启动条件 |

## 风险等级定义

| 等级 | 标记 | 含义 |

# P0 — 致命级（第 1-3 周）

> 不修复则产品无法安全运行。存在资金损失、数据损坏或信任崩塌风险。

---

### P0-1. 修复登录状态刷新丢失

| 维度 | 内容 |
|------|------|
| **问题** | `isAuthenticated` 未加入 Zustand `partialize`，刷新页面后导航栏显示"登录"而非用户头像 |
| **文件** | `frontend/src/stores/auth.ts:54` |
| **方案** | 从 `user !== null && accessToken !== null` 派生 `isAuthenticated`，移除独立状态字段 |
| **开发成本** | 0.5 小时 |
| **收益** | 🔴 **致命** — 所有已登录用户每次刷新都丢失登录态，直接破坏核心体验 |
| **风险** | 🟢 低 — 纯前端状态逻辑改动，不影响 API |

---

### P0-2. 移除首页虚假统计数据

| 维度 | 内容 |
|------|------|
| **问题** | 首页硬编码 "10万+用户"、"100万+图片" 等虚假数据，与画廊实际内容严重矛盾 |
| **文件** | `frontend/src/app/page.tsx:42-47` |
| **方案** | 两个选择：① 删除统计区块 ② 接入真实 API（`/api/users/stats` 全局版）。推荐先删除，后续加真实数据 |
| **开发成本** | 0.5 小时 |
| **收益** | 🔴 **致命** — 任何访问画廊的用户都会发现数据造假，信任彻底崩塌。比赛评委看到会直接扣分 |
| **风险** | 🟢 低 — 删除静态区块，无副作用 |

---

### P0-3. 初始化 Alembic 数据库迁移

| 维度 | 内容 |
|------|------|
| **问题** | `Base.metadata.create_all` 无法修改已有表结构。Alembic 已在 `requirements.txt` 但从未初始化 |
| **文件** | `backend/app/main.py:41`，新建 `backend/alembic/` 目录 |
| **方案** | ① `alembic init` ② 生成初始迁移 ③ 替换 `create_all` 为 `alembic upgrade head` ④ 后续所有 Schema 变更通过 migration 管理 |
| **开发成本** | 1 天 |
| **收益** | 🔴 **致命** — 不解决此问题，后续所有数据库字段变更（如添加 `credits_charged`、`expires_at`）都无法安全上线 |
| **风险** | 🟡 中 — 需要确保现有数据不丢失，建议先备份 SQLite 文件 |

---

### P0-4. 修复头像 URL 路径不匹配

| 维度 | 内容 |
|------|------|
| **问题** | 文件以 `{user_id}_{safe_name}` 保存，但数据库 URL 存为 `/uploads/avatars/{safe_name}`，路径不对应导致头像 404 |
| **文件** | `backend/app/api/users.py:64` |
| **方案** | 修改为 `current_user.avatar_url = f"/uploads/avatars/{user_id}_{safe_name}"` |
| **开发成本** | 10 分钟 |
| **收益** | 🔴 **高** — 头像上传功能完全无效，用户上传头像后显示不了 |
| **风险** | 🟢 低 — 单行改动，但已有头像数据需考虑迁移（如有） |

---

### P0-5. 修复视频双重退款竞态条件

| 维度 | 内容 |
|------|------|
| **问题** | 并发轮询同一视频时，两个请求都可读到 `credits_charged > 0`，导致双重退款 |
| **文件** | `backend/app/services/video_service.py:219-227` |
| **方案** | 方案 A: 在视频记录上加 `SELECT FOR UPDATE` 锁（需 PostgreSQL）。方案 B: 用 `UPDATE ... SET credits_charged = 0 WHERE id = ? AND credits_charged > 0` 原子操作，检查 affected rows |
| **开发成本** | 1 天（含测试） |
| **收益** | 🔴 **致命** — 直接资金损失，每个被并发轮询的视频可能多退一倍积分 |
| **风险** | 🟡 中 — 涉及资金路径，需要编写并发测试验证修复 |

---

### P0-6. 修复孤儿上游任务泄漏

| 维度 | 内容 |
|------|------|
| **问题** | 视频创建时先提交记录再调 Agnes API，若积分不足回滚记录但上游任务已创建，用户不付费但任务在跑 |
| **文件** | `backend/app/services/video_service.py:92-158` |
| **方案** | 调整事务顺序：① 先扣积分（`SELECT FOR UPDATE`）② 再创建视频记录 ③ 再调上游 API。扣费失败直接返回，不创建任务 |
| **开发成本** | 1-2 天（含回归测试） |
| **收益** | 🔴 **致命** — 浪费 Agnes API 配额（RPM ≤ 20 下尤为珍贵），且存在免费生成漏洞 |
| **风险** | 🔴 高 — 重构资金路径核心逻辑，需完整回归测试图片和视频生成流程 |

---

**P0 小计: 6 项，约 4-5 人天**

| 指标 | 估值 |
|------|------|
| 阻塞的问题数 | 3 个致命资金漏洞 + 2 个信任危机 + 1 个架构债 |
| 修复后解锁的能力 | 安全上线、后续所有数据库变更、正确的用户状态 |
| 不修的代价 | 每日资金泄漏 + 用户信任崩塌 + 无法迭代 |

# P1 — 运营级（第 2-6 周）

> 不解决则产品无法正常运营或无法产生收入。

---

### P1-1. 添加路由保护中间件

| 维度 | 内容 |
|------|------|
| **问题** | `/profile`、`/works`、`/keys`、`/topup` 未登录即可访问，显示破损页面 |
| **方案** | 创建 `frontend/src/middleware.ts`，未认证用户重定向到 `/login?redirect=原路径` |
| **开发成本** | 2 小时 |
| **收益** | 🔴 **高** — 消除所有未登录用户看到破损页面的体验 |
| **风险** | 🟢 低 — Next.js 标准模式，不改变业务逻辑 |

---

### P1-2. 添加 Agnes AI 熔断器

| 维度 | 内容 |
|------|------|
| **问题** | Agnes AI 宕机时每个请求等待 360 秒才超时，系统完全卡死 |
| **方案** | 引入 `tenacity` 或 `circuitbreaker` 库：连续 5 次失败 → 熔断 30 秒 → 半开探测 |
| **开发成本** | 1 天 |
| **收益** | 🔴 **高** — Agnes API 不稳定时系统仍能快速失败返回友好提示，而非 6 分钟白屏 |
| **风险** | 🟡 中 — 需要调整 `agnes_ai.py` 的异常处理链，可能影响所有生成流程 |

---

### P1-3. 迁移至 PostgreSQL

| 维度 | 内容 |
|------|------|
| **问题** | SQLite 单写锁 + `SELECT FOR UPDATE` 无效 = 零并发保护。Docker 卷路径也不匹配 |
| **方案** | ① `docker-compose.yml` 添加 PostgreSQL 服务 ② 修改 `DATABASE_URL` ③ 运行 Alembic 迁移 ④ 验证 `FOR UPDATE` 生效 |
| **开发成本** | 2-3 天 |
| **收益** | 🔴 **高** — 解锁并发安全、多 Worker、行级锁。所有积分竞态修复真正生效的前提 |
| **风险** | 🟡 中 — SQLite → PostgreSQL 有类型差异（如 LIKE 大小写敏感），需回归测试 |

---

### P1-4. Token 安全升级（httpOnly Cookie）

| 维度 | 内容 |
|------|------|
| **问题** | Access/Refresh Token 存在 localStorage，XSS 可窃取 30 天有效的 Refresh Token |
| **方案** | Refresh Token 改为 httpOnly Cookie（后端 Set-Cookie），Access Token 仅存内存。前端 `api.ts` 移除 localStorage 读写 |
| **开发成本** | 2 天 |
| **收益** | 🔴 **高** — 消除 XSS 窃取认证凭据的攻击面 |
| **风险** | 🟡 中 — 需要同时改后端（Cookie 设置）和前端（Token 存储逻辑），CORS 配置需调整 |

---

### P1-5. 暴露推荐系统前端 UI

| 维度 | 内容 |
|------|------|
| **问题** | 后端已完整实现推荐码生成和奖励逻辑，但前端零 UI。免费的病毒增长机制完全闲置 |
| **方案** | 在个人中心页添加：推荐码展示 + 复制按钮 + 分享链接 + 已邀请人数 + 获得奖励积分 |
| **开发成本** | 1 天 |
| **收益** | 🟡 **高** — 零成本激活病毒增长飞轮。每带来 1 个新用户，推荐人得 5 积分 |
| **风险** | 🟢 低 — 纯前端新增，后端 API 已就绪 |

---

### P1-6. 暴露图生视频前端 UI

| 维度 | 内容 |
|------|------|
| **问题** | 后端已支持 `mode="i2v"`，但视频生成页无图片上传入口 |
| **方案** | 在视频生成页添加参考图片上传区域（复用图片生成页的 `react-dropzone` 模式），提交时带上 `image_url` |
| **开发成本** | 1-2 天 |
| **收益** | 🟡 **高** — 免费获得一个完整功能。图生视频是当前 AI 视频最热门的使用方式 |
| **风险** | 🟢 低 — 后端已实现，仅添加前端 UI |

---

### P1-7. 画廊分页 / 无限滚动

| 维度 | 内容 |
|------|------|
| **问题** | 画廊只加载 20 条且无分页控件，用户无法浏览更多内容 |
| **方案** | 采用无限滚动（Intersection Observer），每次加载 20 条，滚动到底部自动请求下一页 |
| **开发成本** | 1 天 |
| **收益** | 🟡 **中** — 提升画廊浏览深度，增加内容曝光和用户停留时间 |
| **风险** | 🟢 低 — 前端改动，后端 API 已支持 `page` 参数 |

---

### P1-8. 修复充值页为"即将上线"状态

| 维度 | 内容 |
|------|------|
| **问题** | 充值页 4 个购买按钮完全无响应，用户点击后毫无反馈 |
| **方案** | 付费按钮点击后弹出 Modal："支付功能即将上线，敬请期待"。免费 10 积分按钮保持可用 |
| **开发成本** | 1 小时 |
| **收益** | 🟡 **中** — 从"功能损坏"变为"功能预告"，减少用户困惑 |
| **风险** | 🟢 低 — 纯前端 UI 改动 |

**P1 小计: 8 项，约 9-12 人天**

| 指标 | 估值 |
|------|------|

# P2 — 增长级（第 4-13 周）

> 不推进则产品无法产生收入、无法留住用户、无法建立竞争壁垒。

---

### P2-1. 支付集成（支付宝 + 微信支付）

| 维度 | 内容 |
|------|------|
| **问题** | 产品零收入能力，充值按钮无效 |
| **方案** | ① 后端: 新增 `POST /api/credits/purchase` 端点 ② 集成支付宝/微信支付 SDK ③ 处理支付回调、确认、入账 ④ 前端: 充值页接入真实支付流程 |
| **开发成本** | 2-3 周 |
| **收益** | 🔴 **变革性** — 打开收入通道。按 ¥9.9/50 币定价，1000 付费用户 × ¥30 ARPU = ¥30,000/月 |
| **风险** | 🔴 高 — 涉及资金流转、支付回调、签名验证。需要企业资质、支付商户号。测试需用沙箱环境 |

---

### P2-2. 订阅模式设计

| 维度 | 内容 |
|------|------|
| **问题** | 仅有一次性积分包，无订阅。订阅是收入稳定性和用户留存的核心 |
| **方案** | 三档设计：|

| 档位 | 月费 | 权益 |
|------|------|------|
| 免费版 | ¥0 | 10 积分/月（每月刷新），基础功能 |
| 专业版 | ¥49/月 | 500 积分/月，优先队列，批量生成，超分 |
| 企业版 | ¥299/月 | 无限 API，白标，团队账户，专属客服 |

| 维度 | 内容 |
|------|------|
| **开发成本** | 2 周（与支付集成并行） |
| **收益** | 🔴 **变革性** — 月度经常性收入 (MRR)。100 个专业版用户 = ¥4,900/月 MRR |
| **风险** | 🟡 中 — 需要设计积分刷新逻辑、升降级规则、过期处理 |

---

### P2-3. 每日登录赠送积分

| 维度 | 内容 |
|------|------|
| **问题** | 免费用户 10 积分用完后即流失，无回访理由 |
| **方案** | 每日登录赠送 2 积分，上限累积 20 积分（不登录不累积超过上限） |
| **开发成本** | 半天 |
| **收益** | 🟡 **高** — DAU 提升预计 30-50%。免费用户有持续使用理由，转化为付费的概率增大 |
| **风险** | 🟢 低 — 后端新增每日积分发放逻辑，前端展示"今日已领取"状态 |

| **开发成本** | 1-2 天 |
| **收益** | 🟡 **高** — 注册→首次生成转化率预计从 ~30% 提升至 ~70% |

### P2-10. LLM 驱动的提示词增强
